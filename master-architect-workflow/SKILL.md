# Master Architect Workflow

根据任务复杂度自动路由，避免小任务走冗长流程，大任务不漏关键环节。

---

## 第一步：评估与路由（每次必做）

收到任务后，先做一次快速判断，选择路径并告知用户：

```
[评估] 这是一个 [简单/中等/复杂] 任务，将使用 [快速/标准/完整] 流程。
如需切换，请说"用完整流程"或"直接执行"。
```

**判断标准**:

| 信号 | 路径 |
|------|------|
| 单文件修改、解法明确、无架构影响 | 快速 |
| 多文件改动、存在 2+ 种实现方式、有一定设计决策 | 标准 |
| 跨模块/系统、架构影响、高风险、需要 ADR | 完整 |

---

## 三条路径

### 快速路径

适用于：bug 修复、配置调整、简单 UI 改动

```
直接执行 → [评审] 简短总结
```

- 0 次确认，直接动手
- 不生成文档，除非用户要求

---

### 标准路径

适用于：新功能、第三方集成、中等规模改动

```
[研究+构思] → 确认方案 → [执行] → [评审]
```

**研究+构思阶段**（合并，减少一次停顿）:

1. 读取项目结构，理解现状
2. 如果任务涉及技术选型或全栈实现，且 `fullstack-expert-skill` 可用，参考其技术选项和规范来构建方案
3. 主动提问澄清意图（见下方"意图澄清问题库"）
4. 基于回答提出 **至少 2 个可行方案**，说明各自利弊
5. 等待用户选择方案后，直接进入执行

文档产出：按需，执行后生成 `docs/proposals/NNN-proposal-<任务名>.md`
- 执行完成后，如果 `fullstack-test-automation` 可用，询问用户是否需要补充测试覆盖

---

### 完整路径

适用于：架构设计、系统重构、技术债务、多系统集成

```
[研究] → 确认理解 → [构思] 提问+方案 → 确认方案 → [计划] → 批准 → [执行] → [评审]
```

**研究阶段** `[模式：研究]`:
- 读取项目结构、依赖、相关代码
- 使用 Context7 查询涉及库的最新文档和废弃特性
- 输出：现状理解摘要，等待用户确认

**构思阶段** `[模式：构思]`:
- 主动提问澄清意图（见下方"意图澄清问题库"）
- 如有 `sequential-thinking` 工具，用于复杂推演
- 如果方案涉及技术选型或全栈实现，且 `fullstack-expert-skill` 可用，参考其技术选项和规范
- 提出 **至少 2 个可行方案**，说明利弊
- 将决策记录到 `docs/adr/NNN-adr-<任务名>.md`
- 等待用户选择

**计划阶段** `[模式：计划]`:
- 将选定方案拆解为原子执行步骤
- 明确文件操作（创建/修改/删除）
- 创建 `docs/plans/NNN-plan-<任务名>.md`
- 显式暂停，等待用户回复"执行"或"批准"

**执行阶段** `[模式：执行]`:
- 严格按计划执行，遵循 DRY 和 SOLID 原则
- 生成对应文档（见文档目录规范）
- 如果任务涉及全栈代码实现，且 `fullstack-expert-skill` 可用，参考其技术规范和代码模式执行

**评审阶段** `[模式：评审]`:
- 验证是否符合预期，检查计划完成度
- 提供后续优化建议，询问是否结单
- 如果 `fullstack-test-automation` 可用，询问用户是否需要补充测试覆盖，如需则切换至该 skill

---

## 意图澄清问题库

构思阶段根据任务类型选择 2-4 个最相关的问题主动提问，不要全问。

**通用问题**（几乎每次都问）:
- 这个功能的核心使用场景是什么？有没有边界情况需要特别处理？
- 对性能/可维护性/开发速度，哪个优先级最高？

**架构/设计类**:
- 这个改动未来是否需要扩展？还是一次性的？
- 是否有团队规范或已有模式需要遵循？
- 对第三方依赖的接受程度如何？

**数据/存储类**:
- 数据量级大概是多少？读多还是写多？
- 是否需要考虑数据迁移或向后兼容？

**集成类**:
- 是否有现有的认证/权限体系需要对接？
- 错误处理和降级策略有要求吗？

**重构类**:
- 重构范围是局部还是全局？是否允许破坏性变更？
- 是否有测试覆盖需要维护？

## 方案呈现格式

每个方案必须包含：

```
方案 A：[方案名称]
做法：[一句话描述]
优势：[2-3 点]
劣势/风险：[1-2 点]
适合场景：[什么情况下选这个]

方案 B：[方案名称]
...

推荐：方案 X，原因：[结合用户回答的具体理由]
```

## 文档目录规范

```
docs/
├── adr/              # 架构决策记录（完整路径产出）
├── plans/            # 执行计划（完整路径产出，完成后归档至 plans/archived/）
├── proposals/        # 技术方案（标准/完整路径产出）
├── fixes/            # Bug 修复记录
├── api/              # API 接口文档
├── guides/           # 使用说明和 Demo
└── troubleshooting/  # 故障排查记录
```

### 文件命名规范

所有文档文件名使用 `NNN-<类型缩写>-<任务名>.md` 格式，序号从 `001` 开始，同目录内递增：

| 目录 | 类型缩写 | 示例 |
|------|---------|------|
| `adr/` | `adr` | `001-adr-google-oauth-decision.md` |
| `plans/` | `plan` | `001-plan-add-google-oauth.md` |
| `proposals/` | `proposal` | `001-proposal-google-oauth-impl.md` |
| `fixes/` | `fix` | `001-fix-login-token-expire.md` |
| `api/` | `api` | `001-api-user-endpoints.md` |
| `guides/` | `guide` | `001-guide-oauth-setup.md` |
| `troubleshooting/` | `debug` | `001-debug-oauth-redirect-loop.md` |

**生成文件前**，先扫描目标目录已有文件，取最大序号 +1。目录为空则从 `001` 开始。

## 特殊模式

### 调试模式 `[模式：调试]`

**触发条件**: 用户描述运行时问题、页面异常、交互 Bug
**流程**: Playwright 复现 → 根因分析 → 修复 → 记录到 `docs/troubleshooting/`
**联动**: 如果 `fullstack-test-automation` 可用，修复后建议补充回归测试防止复现

---

## 行为准则

- 使用简体中文，专业简洁，不解释基础概念
- 删除/覆盖操作前必须确认，提供回滚建议
- 重要决策落地为文档，确保可追溯

## 示例

```
用户: "帮我给登录页面增加 Google OAuth。"

[评估] 这是一个中等任务（多文件改动，有方案选择），使用标准流程。

[研究+构思]
当前使用 NextAuth v4，Email 登录已有。

有几个问题想确认：
1. 是否需要保留现有 Email 登录，还是完全替换？
2. 对升级依赖的接受程度？NextAuth v4 稳定但功能有限，Auth.js v5 功能更强但还在 Beta。

(用户回答：保留 Email，不想升级依赖)

方案 A：追加 GoogleProvider（保守）
做法：在现有 [...nextauth].ts 中追加 GoogleProvider 配置
优势：改动最小，风险低，与现有 Email 登录完全兼容
劣势：受限于 NextAuth v4 的功能边界
适合场景：稳定性优先，不打算近期升级

方案 B：迁移至 Auth.js v5（激进）
做法：升级依赖，重写 auth 配置，获得更多 Provider 支持
优势：面向未来，API 更现代
劣势：Beta 版本有破坏性变更风险，迁移成本高
适合场景：计划长期维护，愿意承担升级风险

推荐：方案 A，原因：你明确不想升级依赖，且当前需求方案 A 完全满足。

(用户确认方案 A)

[执行]
已更新 pages/api/auth/[...nextauth].ts
已更新 components/Login.tsx
已生成 docs/proposals/001-proposal-google-oauth.md

[评审]
请在 .env 填入 Google Client ID/Secret。需要我用 Playwright 跑一遍登录流程验证吗？
```
